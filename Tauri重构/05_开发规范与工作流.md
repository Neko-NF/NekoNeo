## 二、日常开发命令

```powershell
# ── 进入项目目录 ──────────────────────────────────────────────────
cd "nekoneo"

# ── 安装依赖 ──────────────────────────────────────────────────────
pnpm install

# ── 启动开发模式（前端 Vite HMR + Rust 热重新编译）─────────────────
pnpm tauri dev

# ── 仅启动前端预览（不需要运行 Rust 后端时）───────────────────────
pnpm dev

# ── 构建生产版本 ──────────────────────────────────────────────────
pnpm tauri build

# ── 运行前端单元测试 ──────────────────────────────────────────────
pnpm test

# ── 运行 Rust 单元测试 ────────────────────────────────────────────
cargo test --manifest-path src-tauri/Cargo.toml

# ── 前端 TypeScript 类型检查 ─────────────────────────────────────
pnpm type-check

# ── Lint 检查 ────────────────────────────────────────────────────
pnpm lint
```

### 热更新说明

| 修改内容 | 效果 |
|---------|------|
| 前端 `.vue` / `.ts` / `.css` | Vite HMR 自动热更新，**无需重启** |
| 后端 Rust 代码（`src-tauri/src/`） | `pnpm tauri dev` 自动重新编译并重启（约 5~15s） |
| `tauri.conf.json` | 需手动重启 `pnpm tauri dev` |

## 三、职责边界（三层禁止混用）

| 层级 | 允许写什么 | 绝对禁止 |
|------|-----------|--------|
| **Views** (`views/*.vue`) | DOM 结构、调用 Composable/Store | 直接 `invoke()`、内联样式、业务逻辑 |
| **Composables** (`composables/*.ts`) | 逻辑封装、调用 `commands.*` | 直接操作 DOM、访问全局 Store（避免循环） |
| **Stores** (`stores/*.ts`) | 全局共享状态、事件监听 | 直接 `invoke()`（应通过 Composable） |
| **Commands** (`src-tauri/commands/*.rs`) | 参数校验、调用 Service | 业务逻辑（下沉到 Service） |
| **Services** (`src-tauri/services/*.rs`) | 核心业务逻辑 | `use tauri::*`（Services 不依赖框架） |

## 四、新增功能标准步骤

### 步骤 1：在 `src/types/` 中定义类型

```typescript
// src/types/example.ts

export interface ExampleData {
  id:    string
  value: number
  label: string
}
```

### 步骤 2：在后端添加 Service 函数

```rust
// src-tauri/src/services/example.rs

pub struct ExampleService;

impl ExampleService {
    pub async fn get_data() -> Result<ExampleData, AppError> {
        // 核心逻辑
    }
}

// 同步编写单元测试
#[cfg(test)]
mod tests {
    use super::*;
    #[tokio::test]
    async fn test_get_data() { ... }
}
```

### 步骤 3：在后端添加 Command

```rust
// src-tauri/src/commands/example.rs

#[tauri::command]
pub async fn example_get_data() -> Result<ExampleData, AppError> {
    ExampleService::get_data().await
}
```

### 步骤 4：注册 Command（main.rs）

```rust
.invoke_handler(tauri::generate_handler![
    // ...已有 commands...
    commands::example::example_get_data,  // ← 新增
])
```

### 步骤 5：在前端封装调用

```typescript
// src/api/commands.ts（追加）
exampleGetData: () => invoke<ExampleData>('example_get_data'),
```

### 步骤 6：编写 Composable

```typescript
// src/composables/useExample.ts

export function useExample() {
  const data = ref<ExampleData | null>(null)
  const loading = ref(false)

  async function fetchData() {
    loading.value = true
    data.value = await commands.exampleGetData()
    loading.value = false
  }

  return { data: readonly(data), loading: readonly(loading), fetchData }
}
```

### 步骤 7：在 View 中使用

```vue
<!-- src/views/ExampleView.vue -->
<script setup lang="ts">
import { onMounted } from 'vue'
import { useExample } from '@/composables/useExample'

const { data, loading, fetchData } = useExample()
onMounted(fetchData)
</script>
```

## 五、新增页面（View）步骤

1. 在 `src/views/` 创建 `NewPage.vue`
2. 在 `src/router/index.ts` 中添加路由：
   ```typescript
   { path: '/new-page', name: 'new-page', component: () => import('@/views/NewPage.vue') }
   ```
3. 在 `src/components/layout/Dock.vue` 的 `navItems` 数组中追加导航项：
   ```typescript
   { name: 'new-page', icon: 'ph-xxx', label: '新页面' }
   ```
   > ⚠️ Dock 导航项数量建议控制在 **5 个以内**，超出会拥挤。如超出，考虑将功能合并到现有页面（如放入设置子分区）。

## 七、设计系统使用规范

### 7.1 颜色使用

```css
/* ✅ 正确：使用 CSS 变量 */
color: var(--text-primary);
background: var(--color-primary-muted);
border-color: var(--border-default);

/* ❌ 错误：硬编码颜色 */
color: #f1f5f9;
background: rgba(6, 182, 212, 0.12);
```

### 7.2 间距使用

```css
/* ✅ 正确：使用间距变量 */
padding: var(--space-3) var(--space-4);
gap: var(--space-2);

/* ❌ 错误：硬编码数字 */
padding: 12px 16px;
gap: 8px;
```

### 7.3 内联样式禁止

```vue
<!-- ❌ 错误：内联颜色/尺寸 -->
<span :style="{ color: '#ef4444', fontSize: '12px' }">错误</span>

<!-- ✅ 正确：使用语义 class -->
<span class="text-danger text-sm">错误</span>
```

## 八、Tauri 权限管理

所有能力（文件访问、系统 API、插件功能）必须在 `src-tauri/capabilities/` 中显式声明，遵循最小权限原则：

```json
// src-tauri/capabilities/default.json

{
  "$schema": "https://schema.tauri.app/config/2",
  "identifier": "default",
  "description": "NekoNeo 默认权限集",
  "windows": ["main"],
  "permissions": [
    "core:default",
    "store:default",
    "autostart:default",
    "single-instance:default",
    "updater:default",
    "notification:default",
    "http:default"
  ]
}
```

> ⚠️ 不允许在 `capabilities` 中添加 `fs:allow-write-all` 等过宽权限，只声明实际使用的细粒度权限。

## 九、版本与发布规范

### 9.1 版本号约定（SemVer）

```
MAJOR.MINOR.PATCH[-CHANNEL.N]

示例：
  2.0.0           # 正式稳定版
  2.0.1           # Patch（仅 Bug 修复）
  2.1.0           # Minor（新功能，向下兼容）
  2.0.0-beta.1    # Beta 测试版
  2.0.0-beta.2    # Beta 迭代
```

### 9.2 更新通道文件（GitHub Releases）

| 通道 | 文件名 | 对应配置 |
|------|--------|---------|
| `stable` | `latest.json` | `updateChannel: 'stable'` |
| `beta`   | `latest-beta.json` | `updateChannel: 'beta'` |

更新描述文件格式（Tauri updater 标准格式）：

```json
{
  "version": "2.1.0",
  "notes": "### 更新内容\n- 优化了 xxx\n\n[MANDATORY] 标记表示强制更新",
  "pub_date": "2026-06-01T00:00:00Z",
  "platforms": {
    "windows-x86_64": {
      "url": "https://github.com/.../NekoNeo-2.1.0-win.exe",
      "signature": "..."
    }
  }
}
```

> 💡 **强制更新标记**：在 `notes` 中包含 `[MANDATORY]` 字符串，前端检测到后弹出不可关闭的强制更新弹窗。

### 9.3 构建产物

```
dist/
├── NekoNeo-2.0.0-win.exe      # NSIS 安装包（主要分发）
├── NekoNeo-2.0.0-win.zip      # 便携版（免安装）
└── NekoNeo-2.0.0-win.exe.sig  # 签名文件（Tauri updater 验证完整性）
```

## 十、测试规范

### 10.1 测试分层

| 类型 | 工具 | 覆盖对象 | 要求 |
|------|------|---------|------|
| **Rust 单元测试** | `cargo test` | Services 层每个 `pub fn` | 关键路径必须有测试 |
| **前端单元测试** | Vitest | Composable、纯工具函数、Store Action | 关键逻辑覆盖 |
| **前端组件测试** | Vitest + @vue/test-utils | 基础组件（NButton、NSwitch 等） | Props / Emits 验证 |

### 10.2 Mock 规范

```typescript
// src/__mocks__/@tauri-apps/api/core.ts
// 测试中 mock 所有 invoke 调用

import { vi } from 'vitest'

export const invoke = vi.fn()

// 在测试文件中覆盖返回值：
// invoke.mockResolvedValueOnce({ running: true, uptimeSec: 120 })
```

### 10.3 CI 流水线（GitHub Actions）

```yaml
# .github/workflows/ci.yml

name: CI
on: [push, pull_request]

jobs:
  test-frontend:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v3
      - run: pnpm install
      - run: pnpm type-check
      - run: pnpm lint
      - run: pnpm test --run

  test-rust:
    runs-on: windows-latest   # Windows-only WinAPI 需要在 Windows 上测
    steps:
      - uses: actions/checkout@v4
      - run: cargo test --manifest-path src-tauri/Cargo.toml

  build:
    needs: [test-frontend, test-rust]
    runs-on: windows-latest
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v3
      - run: pnpm install
      - run: pnpm tauri build
```

## 十一、禁止行为清单

```
后端（Rust）：
❌ 在 Services 层使用 use tauri::*（保持纯 Rust，可独立测试）
❌ 在 Command 层写超过 20 行的业务逻辑（下沉到 Service）
❌ 直接 unwrap() 或 panic!()（使用 ? 配合 AppError）
❌ 使用裸 static mut 变量（使用 Mutex/Arc 包裹）

前端（Vue/TypeScript）：
❌ 在 Views 中直接调用 invoke()（通过 Composable/Store）
❌ 在 .vue 中写内联 style 颜色/间距（使用 CSS 变量+class）
❌ 在 CSS 中硬编码颜色/间距数字
❌ 使用 backdrop-filter: blur（玻璃效果已删除）
❌ 跨组件直接操作 DOM（使用 ref + Vue 响应式系统）
❌ 在 <script setup> 之外写 JavaScript
❌ Store 之间互相导入（通过 Composable 桥接）

通用：
❌ 提交未经 type-check 和 lint 的代码
❌ 在主分支直接 push（通过 PR + review）
❌ 在 capabilities 中声明超出实际需要的权限
```
